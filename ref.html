Hello

The smallest useful program. Save as hello.on and run with oryn hello.on. 

let name = "Oryn"
print("hello from {name}")

Running any .on file requires a package.on marker somewhere in an ancestor directory — an empty file telling the compiler "this is a project root". See Modules.

Comments

Line comments only. Block comments aren't supported.

// a single-line comment
let x = 1 // trailing comment

Primitive types

Four built-in value types plus Range. The type names as they appear in annotations:

Name Description Literal
int 32-bit signed integer 0, 42
float 32-bit float 3.14, 0.5
bool boolean true, false
String UTF-8 string "hi"
Range integer range iterator 0..5, 0..=5

Integer literals are digits only. Float literals must have digits on both sides of the dot — .5 and 5. are not valid. There's no hex, octal, binary, scientific notation, digit separator, or modulo operator. Negative numbers are the unary minus operator applied to a positive literal: -5 parses as -(5).

Parameterized collection types are [T], a homogeneous list, and {K: V}, a homogeneous map — see Lists and Maps.

Bindings

Oryn has two binding forms with deliberately different meanings:

  • let introduces a binding — a name you can rebind to point at different things. Mutable.
  • val introduces a value — a name for one specific thing. Transitively immutable: you can't reassign it, mutate any field, set any indexed element, or call any mutating method on anything reachable from it.

Both forms can optionally carry a type annotation; otherwise the type is inferred from the right-hand side. 

let count = 0         // binding, mutable, inferred as int
count = count + 1  // ok

val pi: float = 3.14  // value, immutable, annotated
// pi = 2.71          // error: cannot reassign val binding `pi`

Reassignment goes through the same = operator — there's no separate :=. Types are checked on every reassignment, so a let declared as int can't later be set to a String.

The val rule reaches all the way through the value: 

obj Bag { items: [int] }

val b = Bag { items: [1, 2] }
// b.items.push(3)    // error: cannot call mutating method `push` on val binding `b`
// b.items[0] = 9     // error: cannot mutate indexed value of val binding `b`
b.items.len()         // ok — len is read-only

Strings

Double-quoted string literals with inline interpolation. Any { } pair inside a string is evaluated as an Oryn expression and stringified. 

let name = "world"
print("hello, {name}")
print("1 + 1 = {1 + 1}")

Supported escape sequences:

  • \\ — a literal backslash
  • \{ — a literal { (skips interpolation)
  • \} — a literal }

Unknown escapes (like \n or \t) are preserved verbatim — the backslash and the next character both end up in the string.

Lists

[T] is a homogeneous, growable sequence of values of type T. It's the only parameterized type in Oryn — the element type is tracked statically at compile time, and [int] and [String] are distinct types that can't be mixed. 

let scores: [int] = [10, 20, 30]
print(scores[0])   // 10

scores[0] = 99     // in-place assignment
print(scores[0])   // 99

List literals use square brackets with comma-separated values. The annotation can be omitted when the element type can be inferred from the literal: 

let names = ["alice", "bob"]   // inferred as [String]
let grid: [[int]] = [[1, 2], [3, 4]]   // nested lists

Empty literals carry no element type on their own, so an annotation is required: 

let xs: [int] = []   // ok — annotation provides context
// let ys = []           // error: cannot infer element type

Methods

Three builtin methods are available on every list, dispatched through the receiver's element type:

Method Signature Notes
len xs.len() -> int current element count
push xs.push(value: T) append in place — value must match the declared element type
pop xs.pop() -> T? remove and return the last element, or nil if empty 
let stack: [int] = []
stack.push(1)
stack.push(2)
print(stack.len())   // 2

if let top = stack.pop() {
    print(top)       // 2
}

Iteration

for … in works directly on lists. The loop variable binds with the element type, so you can access fields and call methods on it without an annotation. 

obj Enemy {
    name: String
    hp: int
}

let squad: [Enemy] = [
    Enemy { name: "orc", hp: 10 },
    Enemy { name: "troll", hp: 25 },
]

for e in squad {
    print("{e.name}: {e.hp}")
}

break and continue work the same as in range loops. Nested iteration falls out naturally — for row in grid { for cell in row { … } }.

Semantics & limits

  • Mutable. Lists are reference values; push, pop, and xs[i] = value mutate in place. A let binding is required if you want to reassign the list itself; val still permits element mutation.
  • Invariant. [int] is not compatible with [int?] or [float]. There's no subtyping between list types.
  • Bounds-checked. xs[i] and xs[i] = value raise a runtime error (list index N out of bounds (length M)) if i is negative or beyond the end of the list.
  • Any element type. Primitives (int, float, bool, String), user-defined obj types, nillables ([int?]), and nested lists ([[int]]) all work. Cross-module obj types need the full qualified path like [math.vec2.Vec2].
  • No equality or slicing yet. Comparing lists with ==, slicing (xs[1..3]), and higher-order methods like map / filter aren't implemented. The orelse and if let forms cover the common pop patterns in the meantime.

Maps

{K: V} is a homogeneous map from keys of type K to values of type V. Key types are limited to String, int, and bool. Values can be any Oryn type. 

let stats: {String: int} = { "hp": 100, "mp": 25 }
print(stats["hp"] orelse 0)   // 100

stats["hp"] = 80              // replace
stats["xp"] = 1200            // insert

Indexing a map returns V?, not V, because the key may be absent. Use orelse or if let to unwrap the result. 

if let hp = stats["hp"] {
    print(hp)
}

print(stats["stamina"] orelse 50)

Empty map literals carry no key or value type on their own, so an annotation is required: 

let flags: {String: bool} = {}   // ok — annotation provides context
// let flags = {}              // error: cannot infer key/value types

Semantics & limits

  • Mutable. Maps are reference values; map[key] = value mutates in place and inserts the key when it is missing.
  • Invariant. {String: int} is not compatible with {String: int?} or {int: int}.
  • No iteration yet. for … in works on ranges and lists, but not maps.
  • No builtin map methods yet. There is no len, remove, or keys method.

Operators

Arithmetic, comparison, logical, and ranges. Listed from tightest binding to loosest — expressions group accordingly without parentheses.

Level Operators Notes
postfix .field, .method(), Type { }, xs[i] left-associative; xs[i] is list or map indexing (see Lists and Maps)
unary -x negation
product *, / no % modulo
sum +, -
range .., ..= exclusive / inclusive
comparison == != < > <= >=
not not x prefix, word keyword
prefix try x, !x error propagation / unwrap-or-trap
and and short-circuit
or or short-circuit
coalesce orelse nil coalescing — loosest binary op

There are no bitwise operators, no compound assignment (+=, -=, …), and no && / || — the logical operators are always spelled and / or / not. The ! symbol is used as the error-unwrap prefix operator and in !T error union type syntax.

Control flow

if / unless / elif / else with required braces, while, and for … in over ranges or lists. Branches and loop bodies are always blocks — no single-statement sugar. 

if grade >= 90 {
    print("A")
} elif grade >= 80 {
    print("B")
} else {
    print("C")
}
unless debug {
    print("debug logging is off")
} else {
    print("debug logging is on")
}
while count > 0 {
    print("{count}")
    count = count - 1
}
for n in 1..=3 {
    print("{n}")
}

for n in 0..10 {
    if n == 3 { continue }
    if n == 6 { break }
    print("{n}")
}

// Lists are also valid iterables — the loop variable is
// bound with the list's element type.
let names: [String] = ["alice", "bob"]
for n in names {
    print(n)
}

elif desugars to else { if … }. unless is the inverse of if: its body runs when the condition is false. There is no match, no ternary, no do-while, and no labels on loops.

Blocks are not expressions in the value sense — let x = { 1 + 2 } does not set x to 3. Use let x = 1 + 2 or call a function.

if let

if let unwraps a nillable value into a branch-scoped binding. See Nil & errors for details. 

let maybe: int? = 42
if let v = maybe {
    print(v)
}

Nil & errors

Oryn has first-class support for absent values and explicit error handling. T? is a nillable type — a value that might be nil. !T is an error union — a value that is either T or an error.

Nillable types

Append ? to any type to make it nillable. nil is assignable to any T?, and a plain T value is also assignable to T? (value promotion). 

let score: int? = 42
let empty: int? = nil

fn maybe_name(id: int) -> String? {
    if id == 1 { rn "Alice" }
    rn nil
}

A T? value cannot be used where T is expected — you must unwrap it first with orelse, if let, or an explicit nil check.

Nil coalescing — orelse

a orelse b evaluates to a if it is not nil, otherwise b. a must be T? and b must be T. The result type is T. 

let x: int? = nil
let y = x orelse 0     // y is int, value is 0

orelse is right-associative, so a chain of fallbacks composes naturally without parentheses. a orelse b orelse c parses as a orelse (b orelse c): every orelse in the chain still sees a non-nillable T on its right (the result of the sub-chain), and only the rightmost operand needs to be plain T as the final fallback. 

let a: int? = nil
let b: int? = 7
let x = a orelse b orelse 0   // x is int, value is 7

Unwrapping with if let

if let tests a T? and, when it is not nil, binds the unwrapped value to a new name scoped to the then-branch. 

let maybe: int? = 7

if let v = maybe {
    print("got {v}")   // v is int, not int?
} else {
    print("nothing")
}

The else branch is optional. The original variable is not narrowed — only the new binding v has the unwrapped type.

Nil equality

== nil and != nil work on any T? value. Comparing two non-nil values still requires them to be the same type. 

let x: int? = 5
print(x == nil)   // false
print(x != nil)   // true

Error unions

!T marks a type that is either a success value of type T or an error. Error unions appear in function signatures to signal that the caller must handle possible failure. 

fn fetch(url: String) -> !String {
    // ...
}

try expr unwraps the success value from an !T. If expr is an error, try immediately returns that error from the enclosing function (which must itself return !U). 

fn process(url: String) -> !int {
    let body = try fetch(url)   // body is String, not !String
    rn 200
}

!expr also unwraps !T to T, but traps (fatal runtime error) if the value is an error. Use it when you're certain success is guaranteed.

Type syntax rules

Nillable and error union modifiers can be combined, but only with parentheses to avoid ambiguity:

Syntax Meaning
int? nillable int
!int error union of int
(!int)? nillable error union — ok
!(int?) error union of nillable — ok
!int? rejected — ambiguous without parens

Functions

Declared with fn. Every parameter requires an explicit type annotation. Return types are optional — omit the -> clause for void functions. 

fn add(a: int, b: int) -> int {
    rn a + b
}

fn greet(name: String) {
    print("hi {name}")
}

fn factorial(n: int) -> int {
    if n <= 1 {
        rn 1
    }
    rn n * factorial(n - 1)
}

rn is how you return a value. Function bodies don't implicitly return their last expression — a function that ends without rn returns void.

Parameter mutability

Parameters are immutable by default. A function cannot reassign a parameter, mutate its fields, set indexed elements on it, or call mutating methods on it. To opt a parameter into mutability, declare it with the mut keyword: 

fn add_item(mut xs: [int], x: int) {
    xs.push(x)         // ok — xs is mut
}

fn sum(xs: [int]) -> int {
    // xs.push(0)         // error: cannot call mutating method `push` on parameter `xs`
    let total = 0
    // ...
    rn total
}

A mut parameter cannot accept a val-rooted argument at the call site — that would let the function mutate something the caller declared immutable. Pass a let binding instead. 

let ys: [int] = [1]
add_item(ys, 2)            // ok

val zs: [int] = [1]
// add_item(zs, 2)         // error: cannot pass to mut parameter 1 of `add_item` from val binding `zs`

Objects

obj is the struct-with-methods type. Fields are name-typed; methods are declared inside the body. A first parameter named self makes the method an instance method; otherwise it's a static method called via TypeName.method(…). 

obj Point {
    x: float
    y: float

    fn length_sq(self) -> float {
        rn self.x * self.x + self.y * self.y
    }

    fn zero() -> Point {
        rn Point { x: 0.0, y: 0.0 }
    }
}

let p = Point { x: 3.0, y: 4.0 }
print(p.length_sq())     // 25.0

let origin = Point.zero()
p.x = 10.0                // fields are assignable on `let` bindings

Object literals use Type { field: value, … }. Every field must be supplied by name — there's no positional construction. Trailing commas are allowed.

Mutating methods (mut self)

Methods that mutate self declare their receiver as mut self, using the same mut keyword that opts non-self parameters into mutability (see Functions). A method with plain self is a read-only view: it cannot write self's fields, call mutating list methods on self's list fields, or call any mut self method on self. 

obj Counter {
    count: int

    // Reads self only — plain `self`.
    fn get(self) -> int {
        rn self.count
    }

    // Writes self.count — receiver must be `mut self`.
    fn bump(mut self) {
        self.count = self.count + 1
    }
}

Calling a mut self method on a val-rooted receiver is a compile error — that's how val's transitive immutability extends through method calls. A plain self method can be called on either let or val bindings. 

let c: Counter = Counter { count: 0 }
c.bump()                  // ok
c.get()                   // ok

val v: Counter = Counter { count: 0 }
// v.bump()              // error: cannot call mutating method `bump` on val binding `v`
v.get()                   // ok — get takes plain self

A method taking plain self may not call a mut self method on self — the plain method's contract said it wouldn't mutate, and delegating to a mutating sibling would break that contract for any val caller relying on it. The same rule applies in reverse: declare the enclosing method's receiver as mut self if it needs to call mutating siblings.

Reassigning self wholesale (self = …) is always a compile error, even inside a mut self method. The local for self is conceptually a receiver borrow; rebinding it would silently no-op for the caller. Mutate self's fields instead, or return a new value and let the caller rebind.

Top-level (non-method) functions don't have a self parameter, so the mut self form doesn't apply. They use the parameter-side mut described in Functions instead — same keyword, same meaning.

Composition with use

Oryn has no inheritance. Instead, use pulls every field and method from another obj directly into the current one. Instances of the composing type carry those fields inline and can call those methods on themselves. 

obj Counter {
    count: int

    fn bump(self) {
        self.count = self.count + 1
    }
}

obj Score {
    use Counter
    label: String

    fn new(label: String) -> Score {
        rn Score { count: 0, label: label }
    }
}

let s = Score.new("points")
s.bump()                      // inherited from Counter
print("{s.label}: {s.count}")

use takes a single type name or a qualified path (use math.shapes.Polygon). Composed fields become part of every Score literal. An obj may have multiple use clauses to compose from several types at once.

Overriding inherited methods

A using type may declare its own method or static method with the same name as one inherited via use. The own version replaces the inherited one in the using type's method table — instances of the using type dispatch to the override. 

obj Health {
    hp: int

    fn tick(self) -> int {
        rn self.hp
    }
}

obj Guard {
    use Health
    name: String

    // Replaces Health.tick on Guard instances.
    fn tick(self) -> int {
        rn self.hp + 10
    }
}

The override must use the same parameter list and the same return type as the inherited method. The one exception is Self-return covariance: a static method that inherited from a type returning that type may instead return the using type. This is what makes overriding constructors like fn make() -> T work cleanly.

The override must also agree with the inherited method's receiver mutability, with one asymmetry: a mut self method may be overridden by a plain self method (the override is more restrictive — strictly safer for val callers, who couldn't call the inherited mutating version but can call the plain override). The reverse — overriding a plain self method with a mut self one — is rejected because it would let mutation flow where the inherited contract said it couldn't, breaking any val caller relying on the parent signature.

There is no super syntax. To call the inherited version from inside the override, give the inherited method a different name and have both versions call it.

Inherited methods do not dispatch through overrides

Inherited method bodies are statically bound at the defining type's compile time. If Health.tick internally calls self.label(), that call always goes to Health.label — even on a Guard instance whose own label would otherwise shadow it. Overrides only kick in when the method is called from outside the inheritance chain.

The reason: there is no virtual dispatch table. The bytecode for Health.tick hard-codes the function index of Health.label at compile time. This keeps method calls cheap and predictable but means you should override the entry point method, not rely on overriding helper methods inside it.

Fields cannot be overridden

Methods can be overridden because they are entries in a name-keyed table; replacing one swaps the table entry. Fields, on the other hand, are slots at fixed offsets in memory: redeclaring an inherited field name would either duplicate storage or shift the layout in a way that breaks every inherited method that reads at the old offset.

A using type that declares a field whose name collides with an inherited field is a compile error. Pick a different name, or wrap the inherited obj as a field instead of composing it via use.

Multi-use conflicts

Two use clauses that contribute the same field or method name conflict by default. The error can be resolved by declaring the same name on the using type itself — the own version replaces both inherited candidates without a signature check, because the user is explicitly picking neither. 

obj Guard {
    use Health
    use Mana

    // Both Health and Mana define `tick(self)`.
    // This explicit declaration resolves the conflict.
    fn tick(self) {
        self.hp = self.hp - 1
        self.mp = self.mp + 1
    }
}

Field name collisions across use clauses are always errors and cannot be resolved this way — the using type would need a different field name anyway, since fields cannot be overridden.

Enums

An enum declares a closed sum type — a value is exactly one of a fixed set of named variants. Variants are either nullary (carry no data) or carry a named-field payload with the same shape as obj fields. Both kinds can mix freely inside the same enum. 

enum Color {
    Red
    Green
    Blue
}

enum GameEvent {
    Damage { amount: int }
    Heal { amount: int }
    Move { dx: int, dy: int }
    GameOver
}

Construct values with EnumName.Variant for nullary variants or EnumName.Variant { field: value, … } for payload variants. Field order in the literal doesn't matter — fields are reordered to declaration order before the value is built. The EnumName. prefix is required: there is no way to import variants directly into the local namespace. 

let c = Color.Red
let e1 = GameEvent.Damage { amount: 10 }
let e2 = GameEvent.Move { dx: 3, dy: 4 }
let e3 = GameEvent.GameOver

Equality is structural: two enum values are equal iff their variant tags match and their payloads are field-by-field equal. There is no pointer/identity comparison — two independently constructed Color.Red values are always equal, and so are two Pair { x: 1, y: 2 } values built at different times. Cross-enum comparisons (a Color against a Mood) are a compile error even when both enums happen to have a variant with the same name — they're distinct types.

match expressions

match dispatches on the active variant. It is an expression, not a statement — every arm body produces a value, and the match's value is the chosen arm's. The result flows into a let binding, a function return, or any other expression context. 

fn rgb_hex(c: Color) -> String {
    rn match c {
        Color.Red   => "#ff0000"
        Color.Green => "#00ff00"
        Color.Blue  => "#0000ff"
    }
}

Patterns are variant paths (Color.Red) or the wildcard _. The wildcard matches anything not covered by a previous arm, so use it to lump "everything else" together. Anything written after a wildcard arm is unreachable and rejected by the compiler. 

fn is_terminal(e: GameEvent) -> bool {
    rn match e {
        GameEvent.GameOver => true
        _                  => false
    }
}

Match expressions enforce two things at compile time:

  • Exhaustiveness — every variant of the scrutinee's enum must appear in some arm, or a _ arm must be present. The compiler lists the missing variants in the error. There's no middle ground: a wildcard switches the check off entirely, but in its absence even one missing variant is an error.
  • Result type uniformity — every arm body must produce the same type. The first arm's type is the canonical one and subsequent arms are checked against it.

Adding a new variant to an enum is a deliberately breaking change: every non-wildcard match on that enum will fail to compile until the new variant is handled. This is the main reason exhaustiveness is enforced — match becomes a tool for finding all the places that care about a particular set of cases.

Slice 1 dispatches on the variant tag only — the payload comes along for the ride but isn't bound into the arm body. Payload destructuring (the Variant { field } => … form that binds payload fields as locals inside the arm) is planned for a future slice.

Visibility

Everything is private by default. The pub modifier opens a declaration up to other modules. It can be applied to:

  • pub let / pub val — top-level constants in a module
  • pub fn — top-level functions
  • pub obj — type definitions
  • pub on a field inside an obj — read/write from outside the module
  • pub fn inside an obj — callable from outside the module

Visibility only bites at module boundaries. Inside a single file, everything can see everything regardless of pub. 

pub obj Color {
    pub r: int
    pub g: int
    pub b: int
    alpha: int              // private — outsiders can't touch it

    pub fn rgb(r: int, g: int, b: int) -> Color {
        rn Color { r: r, g: g, b: b, alpha: 255 }
    }
}

Modules

A project root is any directory containing an empty package.on file. Every other .on file in that tree is a module, addressed by its path relative to the root with slashes replaced by dots and the .on extension dropped. 

project/
├── package.on         // empty marker
├── main.on            // entry point
├── physics.on         // `import physics`
└── math/
    └── vec2.on        // `import math.vec2`

Imports are non-transitive — each file lists what it uses directly. Imported names are always qualified at the call site. 

// main.on
import math.vec2
import physics

let v = math.vec2.Vec2 { x: 3.0, y: 4.0 }
print(physics.fall_distance(1.5))

Module files contain only declarations — import, pub val, pub fn, and pub obj. They don't run top-level code; an importer that needs one of their declarations pulls it in at compile time. pub val and pub let at the top of a module must be literal values (they become compile-time constants).

Testing

Tests are a language-level feature, not a library. A test "name" { … } block at module level declares a named test. The runner discovers them at compile time — no registry, no macro magic — and invokes each one in isolation with a fresh VM state.

assert(cond) is a built-in keyword that requires a boolean expression. A false assertion halts the current test with a diagnostic pointing at the exact expression; the report captures the asserted source text so it reads assertion failed: add(1, 1) == 3 rather than just assertion failed. 

fn add(a: int, b: int) -> int {
    rn a + b
}

test "addition works" {
    assert(add(2, 3) == 5)
    assert(add(0, 0) == 0)
}

test "addition is commutative" {
    assert(add(1, 4) == add(4, 1))
}

Run tests with oryn test [pattern…]. With no argument the runner walks the project root (the nearest ancestor with a package.on) and executes every .on file that contains test blocks. Pass file paths, directories, or glob patterns to narrow the scope. Files without any test blocks are silently skipped. 

$ oryn test

✦ testing · 1 file

    examples/08_tests.on
      ✓ addition works               21µs
      ✓ addition is commutative      2µs
      ✓ max picks the larger value   4µs
      ✗ deliberately broken          1µs

Error: assertion failed: add(1, 1) == 3
    ╭─[examples/08_tests.on:40:12]
 40 │     assert(add(1, 1) == 3)
    │            ───────┬──────
    │                   ╰────── assertion failed: add(1, 1) == 3
────╯
  ✦ 3 passed · 1 failed in 10ms

Tests defined inside an imported module stay dormant unless that module is itself matched by the pattern — oryn test main.on never silently runs tests that live in modules main.on happens to import. This keeps each file's test suite predictable and locally inspectable.

Top-level statements in test files are not executed. Only fn, obj, let/val definitions are made available; the runner invokes test bodies directly, so stray print calls at module scope don't fire.

Builtins

The runtime ships exactly one builtin function:

Name Signature Effect
print print(values…) Writes each value to stdout separated by ", ", then a newline.

Every other function you call comes from your own code or from a module you imported. There is no standard library yet.
